The flags you didn't touch

The lab ran mlx_lm.lora with flags you copied. This deck opens each one: what shapes the adapter, why 4-bit survives, what the loss curve is really saying, and where a silent mismatch ruins a run.

Gourav Shah  ·  School of DevOps & AI  ·  Deep Dive (Part 2)

M3B-DD·01

`r` is how wide the correction notepad is

The frozen base model is a printed textbook. The LoRA adapter is a notepad clipped to each page for corrections — rank is how many lines that notepad gives you.

A wider notepad captures more complex corrections, but it's more paper to carry — and more room to overfit.

M3B-DD·02

Two thin matrices instead of one fat one

LoRA replaces a weight update to W (d×d) with two small matrices A (d×r) and B (r×d). Trained parameter count scales with r, not with d².

Trainable parameters scale with r, not with d² — this is why rank is the primary memory/capacity dial.

M3B-DD·03

`alpha/r` is a ratio, not two knobs

The effective update is (alpha/r) × (A×B), not just A×B. Two configs at different widths can carry the exact same effective step size.

Bump the rank without rescaling alpha and you've silently changed every adapted layer's step size too.

M3B-DD·04

Not every layer gets a notepad

target_modules picks which weight matrices get adapted. Query/value projections shape what the model attends to — the default starting point.

Attention's query/value projections are the highest-leverage place to start — not the only place.

M3B-DD·05

Dropout stops the notepad memorizing example order

During training, dropout randomly zeroes a fraction of the adapter's activations each step, so the adapter can't over-rely on any one path.

Matters little on 8 toy examples — it's what keeps a real 500–2000-example adapter from memorizing example order instead of the pattern.

M3B-DD·06

16 buckets, placed where the weights actually live

Neural network weights cluster near zero with thin tails. NF4 places its 16 representable values to match that bell curve, not spread evenly across the range.

Weights are normally distributed, so NF4 spends its 16 values where the data is — that's why 4-bit survives here.

M3B-DD·07

Double quantization: quantizing the quantizer's own numbers

NF4 needs small per-block scaling constants to map weights into its buckets. Those constants add up across a whole model — so double quantization quantizes them too.

Small per-weight, but it adds up — roughly another 0.4 bits per parameter at 7B+ scale.

M3B-DD·08

Storage in 4-bit, math in 16-bit

The frozen base sits in NF4 at rest — that's the memory saving. It's dequantized on the fly to bf16/fp16 for the actual matmul; the adapter itself never touches 4-bit.

The part that needs gradient precision — the adapter — never touches 4-bit; that's why QLoRA trains stably.

M3B-DD·09

The learning rate is the optimizer's step size

Too high and loss oscillates or diverges; too low and 50 iterations won't move the needle. The lab's default is deliberately conservative for a tiny run.

Production configs run higher than the lab default, but pair it with a decay schedule to start fast and settle gently.

M3B-DD·10

Falling train loss can still mean the adapter is memorizing

A healthy curve keeps falling on train and validation together. The memorization signature is train loss falling while validation stalls or rises.

Before trusting "loss went to near-zero," ask for the validation curve and a held-out generation — not just the train number.

M3B-DD·11

Bigger batch costs memory; more accumulation steps don't

A larger batch gives a more stable gradient but costs proportionally more memory. Gradient accumulation simulates a larger batch without the memory cost.

On a memory-constrained machine, reach for gradient accumulation before a bigger --batch-size — that's the lever that actually blows your budget.

M3B-DD·12

Same JSON shape, wrong token boundaries — and it still trains

Every instruct model has a chat template baked into its tokenizer. If your training data's shape doesn't match it, training runs and loss falls — but silently on the wrong signal.

A format mismatch doesn't error — it silently trains the adapter on the wrong signal, and it underperforms in the field.

M3B-DD·13

One fixed prompt set, reused every time you change a knob

Generation is stochastic — one good-looking output proves nothing. Compare variants side by side on the same fixed held-out prompts, every time.

Only a repeated fixed comparison shows a real behavior change — a single generation proves nothing.

M3B-DD·14

Missing knowledge or missing behavior — the fork before you fine-tune

A bad prompt costs an edit. A bad fine-tune costs a training run and a re-evaluation. Reach for fine-tuning only after prompting and RAG are exhausted.

A bad prompt costs an edit; a bad fine-tune costs a training run — exhaust prompting and RAG first.

M3B-DD·15

Same lab, three adapters, one comparison table

Baseline (rank 8, lr 1e-5), Variant A (rank 4), Variant B (lr 1e-4) — same eight training rows, each written to its own adapter directory.

Every variant trains on the identical eight rows — only the rank or the learning rate changes, isolating one knob at a time.

M3B-DD·16

Cutting rank in half measurably raises the floor

Rank 8 (baseline) lands at train loss 0.200. Rank 4 (Variant A), same data and learning rate, converges to a measurably higher 0.449.

The capacity cut is real and visible even on eight examples — confirms rank is the primary capacity dial.

M3B-DD·17

10x the learning rate, and no instability shows up

Variant B (10x learning rate) converged faster and lower — 0.054 vs baseline's 0.200 — with a smooth monotonic curve, no oscillation.

A suspiciously good loss curve on a trivial dataset tells you about the dataset's easiness, not the learning rate's safety at production scale.
