## Five real incompatibilities found & fixed (reuse repo assumed Docker Desktop / x86)

1. **`range of CPUs is from 0.01 to 2.00`** — compose `cpus: 4.0` cap exceeded the VM's 2 CPUs.
   Fixed by allocating 4 CPUs / 6 GB to the runtime VM (documented as a lab setup step).
2. **`Too large swap space. 4.00 GiB out of the 3.82 GiB total CPU memory`** — vLLM `--swap-space`
   defaults to 4 GiB. Fixed with `--swap-space 1`.
3. **`numa_migrate_pages failed. errno: 1`** (EPERM) — CPU backend migrates NUMA pages at startup.
   Fixed with `cap_add: [SYS_NICE]` + `security_opt: ["seccomp:unconfined"]`.
4. **HTTP 500 `rms_norm_impl not implemented for 'BFloat16'`** — `--dtype auto` chose bf16, which CPU
   kernels can't run on arm64. Fixed with `--dtype float32`.
5. **float32 360M crash-looped** (KV cache didn't fit the 6 GB VM). Fixed by defaulting to the lighter
   **135M** model + `--max-model-len 1024` / `--max-num-seqs 4`.

## checks.json validation 2026-07-22

`up.sh` checks `docker info` reports >= 4 CPU / >= 6 GB before building, then runs

### Environmental finding: stale Rancher Desktop port-forward on 8009 (this machine only)

During this validation session, host port **8009** (the lab's documented default,
`VLLM_PORT:-8009` in `compose.yaml`) returned `Recv failure: Connection reset by peer` for every
request, on a fresh container, on a restarted container, and even with **no container at all**
bound to that port. The vLLM server itself was confirmed fully healthy and answering
(`docker exec vllm-smollm2 curl localhost:8000/v1/models` → real JSON, `docker inspect` →
`healthy`); only the host-side forwarder for port 8009 specifically was broken. Other host-forwarded
ports on the same Rancher Desktop VM (`11434` native Ollama, `5100` an unrelated registry) worked
fine throughout, and switching to `VLLM_PORT=8010` fixed it instantly with zero other changes —
confirming this is local Rancher Desktop port-forwarder state on this dev machine, not a defect in
`compose.yaml`, `up.sh`, or the lab's default port choice. **No change made to the lab's documented
default (`8009`)** — a learner on a fresh Rancher Desktop install will not hit this. Live validation
below ran with `VLLM_PORT=8010` to prove the pipeline end-to-end; `checks.json` and `up.sh` ship
unchanged with the `8009` default.

A second contributing factor found during debugging: an unrelated pre-existing container on this
machine (`opsmate-control-plane`, a `kind` k8s node from other work) was consuming 5.9 GiB of the
shared 7.7 GB Rancher Desktop VM, which slowed vLLM's first startup after a restart. Stopped it for
the validation run and restarted it afterward — not a lab defect, just this machine's other
workloads competing for the shared VM (reinforces the course's "one module at a time" discipline).

## 2026-07-22 — post-review fix: port parameterized; re-validated

`VLLM_PORT`. Replaced the literal `8009` with `${VLLM_PORT:-8009}` in `v1-models-smollm2`,
`chat-completion-choices`, and `health-endpoint` so the checks and `up.sh` can never disagree on
port again.
