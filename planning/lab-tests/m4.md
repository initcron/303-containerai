# Lab-test evidence — M4: Packaging Models as OCI Artifacts (KitOps)

**Machine:** Apple Silicon (arm64), 16 GB RAM · Rancher Desktop · **kit v1.15.0**
**Date:** 2026-07-05
**Artifact:** `acme-docs-model:1.0.0` = SmolLM2-135M GGUF (100.6 MB) + `prompts.txt` config

## Install (documented in the lab)

`kitops` is not a core Homebrew formula and the `jozu-ai/kitops` tap is "untrusted". The reliable install
is the release binary:

```bash
curl -fsSL https://github.com/kitops-ml/kitops/releases/download/v1.15.0/kitops-darwin-arm64.tar.gz \
  | tar xz -C /tmp && sudo mv /tmp/kit /usr/local/bin/kit    # or any dir on PATH
kit version    # Version: 1.15.0
```

(Brew alt: `brew tap kitops-ml/kitops && brew trust --formula kitops-ml/kitops/kitops && brew install kitops`.)

## Push / pull / unpack — validated against a local OCI registry

GHCR push returned `denied: permission_denied: token ... scopes` — the `gh` token lacks **`write:packages`**
(a real GHCR requirement; enable via `gh auth refresh -h github.com -s write:packages` or a PAT, and make
the package public). To validate the KitOps mechanics live, we used a local `registry:2` (identical flow,
`--plain-http`):

```
$ docker run -d -p 5001:5000 --name m4-registry registry:2
$ kit tag ghcr.io/initcron/acme-docs-model:1.0.0 localhost:5001/acme-docs-model:1.0.0
$ kit push --plain-http localhost:5001/acme-docs-model:1.0.0
Pushed sha256:a72965fa...

# simulate a clean machine
$ kit remove localhost:5001/acme-docs-model:1.0.0
$ kit unpack --plain-http localhost:5001/acme-docs-model:1.0.0 -d /tmp/m4-clean
Unpacking config to Kitfile / model to ./model/...gguf / code to ./prompts.txt
$ ls -lh /tmp/m4-clean/model/*.gguf   ->  100.6M   (byte-identical to what was pushed)
```

## Selective pull — the KitOps payoff

Fetch only the layer you need with `--filter` (types: `model`, `code`, `docs`, `prompts`, `datasets`):

```
$ kit unpack localhost:5001/acme-docs-model:1.0.0 --filter=model -d ./weights-only
# grabs the model layer without the code/config — e.g. serving nodes pull weights, not datasets
```

## Verdict

✅ KitOps packs a model + config into an OCI ModelKit, pushes it to any OCI registry, and pulls/unpacks it
on a clean environment with byte-identical contents — plus selective per-layer pull. GHCR is the taught
target; it just needs a `write:packages` token + a public package.

## checks.json validation 2026-07-22

**Assets added:** `labs/m4/checks.json`, `labs/m4/up.sh`, `labs/m4/down.sh`.

`up.sh` starts a local `registry:2` container (`docker run -d -p 5001:5000 --name m4-registry
registry:2`, idempotent — removes any stale container from a prior run first) and polls
`GET /v2/` until it answers (60s timeout, 2s between polls). `down.sh` force-removes the container
(idempotent — safe to run when nothing is up).

**Kept the lab's documented host port (`5001`, mapping to the registry's internal `5000`)** —
that's what `labs/m4/README.md`/lab.md Step 5b and the earlier evidence above already validated
live; matches "local `registry:2` on :5000" from the task brief (the container's own listen port).

The checks pack/push/pull round-trip does **not** require the learner's ~100 MB GGUF download —
it builds a small synthetic Kitfile + dummy weight file in a `mktemp -d` scratch dir (cleaned up via
`trap`), so the assertion (pack -> push -> remove local -> unpack from registry -> `kit list`
confirms the tag survived) runs in seconds and needs no network access beyond the local registry.
The real ~100 MB SmolLM2 GGUF flow is exercised by the learner following the lab prose itself; the
checks assert the *mechanics* KitOps guarantees (byte-identical round trip through an OCI registry),
which is the module's actual point.
