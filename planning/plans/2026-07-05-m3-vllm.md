**Goal:** Teach production-grade serving with **vLLM** behind the *same* OpenAI-compatible `/v1` endpoint

track is documented. Learn PagedAttention, continuous batching, quantization, and CPU tuning.

form. Validate on Rancher Desktop (`PATH="$HOME/.rd/bin:$PATH"` for automated docker).

linux/arm64** → runs natively on this Mac. Serves `HuggingFaceTB/SmolLM2-360M-Instruct` on an
OpenAI-compatible server at container `:8000` (host `:8009`), `/health` + `/v1/...`. A live build+run was
kicked off during planning; capture its real output for the lab evidence.

## Key facts from the reuse repo (adapt, don't reinvent)
- **Dockerfile:** `FROM openeuler/vllm-cpu:0.9.1-oe2403lts` + a one-line NUMA patch (sed on
  `cpu_worker.py` to handle zero NUMA nodes in containers). This patch is the repo's signature teaching
  point — explain WHY (containers report 0 NUMA nodes → division by zero without the guard).
- **compose:** service `vllm-cpu`, command `--model … --host 0.0.0.0 --port 8000 --dtype auto
  --max-model-len 2048 --max-num-seqs 8 --trust-remote-code`; env `VLLM_TARGET_DEVICE=cpu`,
  `VLLM_CPU_KVCACHE_SPACE`, `OMP_NUM_THREADS=2`, `OPENBLAS/MKL_NUM_THREADS=1`; `deploy.resources.limits`
  cpus 4 / mem 8G; healthcheck `curl /health`; port `8009:8000`; `hf-cache` volume.
- **Model presets** (`.env`): SmolLM2 135M / 360M / 1.7B — 360M is the balanced default. Use 360M for the
  live lab; mention 135M for the lightest machines.

## Task 1 — Lesson (`site/docs/m3-vllm/lesson.md`)
Analogies + ≥1 Mermaid. Sections:
1. **Why vLLM for production** — PagedAttention + continuous batching → ~3× throughput over Ollama under
   load. Analogy: Ollama = a home espresso machine (great for one cup); vLLM = a commercial machine with
   a queue system that pulls many shots in parallel without idle time (continuous batching). Or:
   PagedAttention = virtual memory paging for the KV-cache (no wasted contiguous reservation).
2. **Same contract, bigger engine** — vLLM exposes the SAME OpenAI-compatible `/v1` endpoint as Ollama
   (M2). Swapping Ollama→vLLM changes nothing in the client. Mermaid: M2 client → `/v1` → {Ollama |
   vLLM} — reinforce the universal-contract idea from M2.
3. **CPU track (universal, what you'll run)** — why a CPU image exists: to learn the OpenAI server,
   batching, and quantization mechanics on any laptop. The `openeuler/vllm-cpu` image + the NUMA patch;
   thread tuning (`OMP_NUM_THREADS`, `VLLM_CPU_KVCACHE_SPACE`) that keeps a laptop responsive. It's slow
   on purpose — throughput isn't the point here, understanding is.
4. **GPU track (throughput)** — `vllm/vllm-openai` image + NVIDIA Container Toolkit (`--gpus all`,
   `--ipc=host`), safetensors, VRAM sizing; TGI as an alternative. Documented, not run here (Apple GPU
   reality from Setup). A small Mermaid or table: CPU-learn vs GPU-serve.
5. **Quantization in practice** — AWQ vs GPTQ vs FP8: the accuracy/throughput trade-off (brief table).
6. **Operational gotchas** — `--ipc=host`/shared memory, `--max-model-len`/`--max-num-seqs`, VRAM/RAM
   sizing, the NUMA patch. ~1000–1300 words. End: "In the lab you'll serve SmolLM2 on CPU vLLM and hit
   the same `/v1` your M2 client already speaks."

## Task 2 — Lab (`site/docs/m3-vllm/lab.md`) + assets
Copy-runnable, each command + **Expected output**. Plain `docker`/`docker compose` in prose. Warn that
first run pulls a multi-GB image + downloads the model (cached after) and that CPU inference is slow.
1. **The image + the NUMA patch** — show `labs/m3/Dockerfile` (openeuler base + sed patch); explain the
   patch. `docker compose build`.
2. **Serve SmolLM2 on CPU vLLM** — `labs/m3/compose.yaml` (the `vllm-cpu` service, resource caps, thread
   env, port 8009). `docker compose up -d`; wait for health: `curl http://localhost:8009/health`.
3. **It's the same OpenAI contract** — `curl http://localhost:8009/v1/models` and
   `.../v1/chat/completions`; then point M2's client at it by changing only `OPENAI_BASE_URL` to
   `http://host.docker.internal:8009/v1` — same client, new engine. (Reinforce the through-line.)
4. **Tuning knobs** — try `MAX_NUM_SEQS`, `OMP_THREADS`, a smaller 135M preset via `.env`; observe.
5. **GPU track (read-only)** — the `vllm/vllm-openai` + `--gpus all` commands to run on an NVIDIA box /
   cloud VM (not executed here).
6. **Troubleshooting** — slow first token (CPU, expected), OOM (lower `--max-model-len`/mem), model
   download stalls, NUMA error if patch missing.
7. **Tear down** — `docker compose down` (free RAM before next module). What's next: M4 packages models
   as OCI artifacts.
Also `labs/m3/.env.example` (presets) + `labs/m3/README.md`.
