# M2 · Serving Local Models the Universal Way — Plan

> Follows the proven M1 pattern (lesson w/ analogies + Mermaid → lab → live validation → quiz).
> One implementer wave + controller review + push/deploy.

**Goal:** Teach the **OpenAI-compatible endpoint as the universal serving contract** — swap engines
without touching app code — and have the learner containerize a small Python client that calls the model,
starting the hand-authored `compose.yaml`.

**Constraints:** inherit `CLAUDE.md`. Model native (Ollama) at `host.docker.internal:11434`; dev model
`qwen2.5:1.5b`. Validate on Rancher Desktop (`PATH="$HOME/.rd/bin:$PATH"` for automated docker).

**Verified de-risk (2026-07-05):** Ollama exposes OpenAI-compatible `/v1/models` and
`/v1/chat/completions`; both work from host and from a container via `host.docker.internal:11434/v1`
(returns a valid `choices[0].message.content`). API key can be any dummy string (e.g. `ollama`).

## Task 2 — Lab (`site/docs/m2-serving/lab.md`) + assets
Copy-runnable, each command + **Expected output** block. Plain `docker` in prose.
1. **Hit the OpenAI-compatible endpoint directly** — `curl .../v1/models` and `.../v1/chat/completions`
   (host, then from a throwaway container via `host.docker.internal`). Contrast with M1's native
   `/api/generate`.
2. **Write the client** — `labs/m2/client.py` using the `openai` SDK:
   ```python
   import os
   from openai import OpenAI
   client = OpenAI(base_url=os.environ.get("OPENAI_BASE_URL", "http://host.docker.internal:11434/v1"),
                   api_key=os.environ.get("OPENAI_API_KEY", "ollama"))
   resp = client.chat.completions.create(
       model=os.environ.get("MODEL", "qwen2.5:1.5b"),
       messages=[{"role": "user", "content": "Explain containers in one sentence."}])
   print(resp.choices[0].message.content)
   ```
3. **Containerize it** — `labs/m2/Dockerfile`:
   ```dockerfile
   FROM python:3.12-slim
   RUN pip install --no-cache-dir openai
   WORKDIR /app
   COPY client.py .
   CMD ["python", "client.py"]
   ```
4. **Start the growing `compose.yaml`** — `labs/m2/compose.yaml` with ONE service (the client) that
   the learner hand-authors; it reaches the native model via `host.docker.internal`:
   ```yaml
   services:
     client:
       build: .
       environment:
         OPENAI_BASE_URL: http://host.docker.internal:11434/v1
         MODEL: qwen2.5:1.5b
       # extra_hosts not needed on Rancher Desktop/Docker Desktop; note it for plain Linux:
       # extra_hosts: ["host.docker.internal:host-gateway"]
   ```
   Run: `docker compose run --rm client` → prints the model's answer. Explain each block (this file GROWS
   across modules — model stays native/shared).
5. **Prove engine-swappability (concept)** — change only `OPENAI_BASE_URL` to another OpenAI-compatible
   engine; client code unchanged.
6. **Troubleshooting** — connection refused (`OLLAMA_HOST=0.0.0.0`), `host.docker.internal` on plain
   Linux needs `extra_hosts: host-gateway`, dummy API key required by the SDK.
7. **What's next** — M3 swaps Ollama for vLLM behind the *same* endpoint.
Also `labs/m2/README.md`.
